# 餐饮推荐增强与地图分类标记设计

## 背景

当前行程页已经具备基础的 AI 行程生成、地图展示和推荐住宿/餐饮能力，但还存在两个明显问题：

1. 餐饮推荐只有一组列表，且主要围绕目的地中心点搜索，缺少“景点附近顺路吃”和“专门值得去吃”的区分。
2. 地图点位只区分住宿、美食、景点三类，无法清晰表达停车、加油、充电等自驾补给点。

本次设计要解决这两个问题，同时保持现有历史行程、地图联动和导出能力不被破坏。

## 目标

1. 将餐饮推荐拆成 `景点附近` 和 `爆火推荐` 两组。
2. 在 `Self-drive` 模式下补充 `停车`、`加油`、`充电` 推荐点。
3. 为 `景点`、`住宿`、`吃饭`、`停车`、`加油`、`充电` 提供稳定且易识别的地图图标。
4. 保持推荐数据以后端为主，保证新生成行程和历史行程展示一致。
5. 继续兼容旧字段，避免已有前端或历史数据直接失效。

## 非目标

1. 本次不引入新的第三方 POI 数据源，仍以高德接口为主。
2. 本次不重做整个页面布局，只调整推荐区块和地图标记体系。
3. 本次不做“实时无限刷新附近推荐”，避免前端和后端出现双重搜索逻辑。

## 选型结论

采用“后端统一产出推荐结果，前端负责展示和联动”的方案。

原因：

1. 后端可以基于完整行程上下文统一计算推荐，保证数据源单一。
2. 历史行程可以完整复现，不依赖前端临时再次请求高德。
3. 前端仍可保留现有坐标补全机制，但不承担推荐逻辑本身。

未采用的方案：

1. 前端在切换天数时临时搜索附近吃饭和补给点：交互看似更灵活，但会导致数据不可复现，且更容易受前端 Key、配额和网络抖动影响。
2. 仅增加餐饮数量、不调整数据结构：实现快，但无法满足“景点附近 + 爆火推荐”的明确需求。

## 现状约束

### 当前后端

1. `TripPlannerService.planTrip()` 先生成 AI 行程草稿，再调用 `PlaceDiscoveryService.discover(destination)` 获取住宿和餐饮推荐。
2. `PlaceDiscoveryService` 目前只围绕目的地中心点搜索 `酒店` 和 `餐厅/美食`。
3. `TripPlanResponse` 当前只有 `recommended_hotels` 和 `recommended_restaurants` 两组推荐字段。

### 当前前端

1. `frontend/app.jsx` 只渲染住宿和美食推荐卡片。
2. AMap 标记类型目前只有 `stay`、`food`、`spot`、`other`。
3. 地图点位数据主要来自 `recommended_hotels`、`recommended_restaurants` 和 `daily_itinerary.activities`。

## 设计总览

### 数据流

1. `TripPlannerService` 先生成 `AiTripDraft`。
2. `TripPlannerService` 将目的地、交通方式和已生成的行程节点传入 `PlaceDiscoveryService`。
3. `PlaceDiscoveryService` 依据路线锚点和交通方式产出扩展后的推荐结果。
4. `TripPlanResponse` 返回新的推荐字段，同时保留兼容字段。
5. 前端按推荐分组渲染列表，并将所有分类点位汇总到地图。

### 职责划分

1. `TripPlannerService`
   - 负责组织流程。
   - 在 AI 行程生成后，把行程活动坐标传给地点发现层。

2. `PlaceDiscoveryService`
   - 负责根据目的地、交通方式、路线锚点生成住宿、餐饮和自驾补给推荐。
   - 负责分项降级，不让单个查询失败拖垮整包推荐。

3. 前端
   - 负责分组渲染推荐区块。
   - 负责地图图标、图例和点击联动。
   - 继续保留已有 AMap 前端补点能力，用于坐标缺失时的兜底。

## 后端设计

### 接口调整

`PlaceDiscoveryService.discover()` 需要从只接收 `destination`，调整为接收完整的推荐上下文，建议形态如下：

1. `destination`
2. `travelMode`
3. `dailyItinerary` 或可解析为路线锚点的等价结构

这样地点发现层可以直接基于已生成的景点坐标做附近搜索，而不是只围绕目的地中心点。

### 新增返回字段

在 `TripPlanResponse` 中新增以下字段：

1. `recommended_food_nearby`
2. `recommended_food_hot`
3. `recommended_parking`
4. `recommended_refuel`
5. `recommended_charging`

兼容策略：

1. `recommended_restaurants` 保留一版，作为兼容汇总字段。
2. 新前端优先读取新字段。
3. 旧前端若仍读取 `recommended_restaurants`，也能拿到基础餐饮结果，不至于直接报错。

### 推荐算法

#### 住宿

保留现有围绕目的地中心点的搜索方式，不作为本次重点改动项。

#### 景点附近餐饮

`recommended_food_nearby` 的生成规则：

1. 从每天行程中选择最多 `2-3` 个核心活动作为搜索锚点。
2. 锚点优先使用有经纬度的景点活动，跳过明显泛化的占位活动。
3. 对每个锚点执行附近餐饮搜索，优先保留距离近、非连锁、非快餐点位。
4. 全部结果去重后按“距锚点最近”优先排序。
5. 产出整趟行程总表，前端再根据当前选择的天数筛出当天附近推荐。

为了支持前端按天联动，推荐点需要额外带出其关联的 `day` 信息，或后端至少提供可供前端还原归属的路线关联信息。

#### 爆火推荐

`recommended_food_hot` 的生成规则：

1. 围绕目的地中心点或核心城区进行较大范围搜索。
2. 关键词组合使用：`餐厅`、`美食`、`本地菜`、`必吃`、`热门餐厅`。
3. 去除明显低质项、快餐和强连锁品牌。
4. 不强求距离最近，优先保留更有“目的地代表性”的候选。
5. 最终列表稳定展示，不随当天切换而变化。

#### 自驾补给

仅当 `travelMode = Self-drive` 时生成：

1. `recommended_parking`
2. `recommended_refuel`
3. `recommended_charging`

排序策略：

1. 优先离当天路线近。
2. 其次离当天首个或末个主要景点近。
3. 如果行程缺少足够坐标，再回落到目的地中心点搜索。

非自驾模式下，这三组字段统一返回空列表。

### 降级策略

降级必须按类别独立进行：

1. `景点附近餐饮` 查询失败：回落到目的地中心附近餐饮。
2. `爆火推荐` 查询失败：不影响 `景点附近` 和住宿结果。
3. `停车/加油/充电` 任一类别失败：仅清空该类别，不影响其他类别。
4. 坐标缺失：尽量保留名称和地址，让前端现有 AMap 补点机制继续兜底。
5. 如果某类完全没有结果：返回空数组，不伪造具体热门店名称。

### 数据结构建议

推荐使用统一的 `PlaceRecommendation` 结构继续承载各类点位，并按需增加轻量扩展字段：

1. `category`
2. `address`
3. `reason`
4. `source`
5. `source_url`
6. `latitude`
7. `longitude`
8. `day` 或等价的路线归属标识

如果不希望直接修改 `PlaceRecommendation`，也可以新增专用于地图和推荐联动的轻量字段对象，但优先建议复用同一结构，减少序列化和前端处理分支。

## 前端设计

### 推荐区块

推荐区块调整为四个模块：

1. `住宿推荐`
2. `景点附近`
3. `爆火推荐`
4. `自驾补给`

展示行为：

1. `景点附近` 跟随当前选中的天数切换。
2. `爆火推荐` 作为整趟行程稳定展示，不因切换天数而变化。
3. `自驾补给` 仅在 `travel_mode = Self-drive` 时展示。
4. 某个模块没有结果时直接隐藏，不显示空白骨架。

### 地图图标

地图图标改为统一样式、不同语义标识：

1. 景点：`景`
2. 住宿：`宿`
3. 吃饭：`食`
4. 停车：`P`
5. 加油：`油`
6. 充电：`电`

颜色建议：

1. 景点：青绿色
2. 住宿：深蓝
3. 吃饭：橙红
4. 停车：靛蓝
5. 加油：暖棕
6. 充电：亮青

前端映射层需要扩展 `kind`：

1. `spot`
2. `stay`
3. `food`
4. `parking`
5. `refuel`
6. `charging`
7. `other`

### 地图图例

在地图区域添加固定图例卡片，展示当前所有可见图标类别及其文案，避免用户依赖猜测理解颜色含义。

### 联动行为

1. 点击某一天行程：
   - 地图切换到当天路线。
   - `景点附近` 同步切换为该天附近餐饮。
   - 自驾模式下，自驾补给优先突出与该天路线关联最近的点位。

2. 点击任意推荐项：
   - 地图直接定位到该点。
   - 自动打开信息窗。
   - 列表项进入高亮态。

3. 点击同一天按钮且当前为堆叠布局：
   - 仍保持现有“切到地图区”的行为。

### 兼容与降级

1. 新前端优先读新字段。
2. 如果历史数据只有 `recommended_restaurants`，前端可将其作为 `爆火推荐` 或回退餐饮总表展示，避免旧数据页面空白。
3. 某类点位无坐标时，继续复用现有 AMap PlaceSearch / Geocoder 补点逻辑。

## 测试设计

### 后端测试

至少覆盖以下场景：

1. 非自驾模式下，`recommended_parking`、`recommended_refuel`、`recommended_charging` 为空。
2. 自驾模式下，会生成补给点。
3. 餐饮被正确拆成 `recommended_food_nearby` 和 `recommended_food_hot`。
4. `景点附近餐饮` 优先使用行程景点坐标，而不是只使用目的地中心点。
5. 单个高德类别查询失败时，不会让整包推荐失败。
6. 兼容字段 `recommended_restaurants` 仍能生成稳定结果。

### 前端测试

至少覆盖以下场景：

1. 推荐区块能渲染新的四个模块。
2. 非自驾模式下不渲染 `自驾补给`。
3. 地图点位收集逻辑能识别 `parking`、`refuel`、`charging`。
4. 图标渲染函数能映射到正确的中文字标和颜色。
5. 选中某一天后，`景点附近` 会随天数联动切换。
6. 老数据只有 `recommended_restaurants` 时页面仍能正常展示。

### 不在本次范围内

1. 不改 OpenAI 行程生成提示词的主策略。
2. 不新增后端数据库结构变更。
3. 不做地图供应商切换。

## 风险与对策

1. 高德搜索结果质量不稳定。
   - 对策：做类别级降级和兼容展示，不把单点失败升级为整页失败。

2. 历史数据缺少新字段。
   - 对策：前端保留 `recommended_restaurants` 兼容读取路径。

3. 路线锚点坐标不足导致“景点附近”质量下降。
   - 对策：优先使用已有坐标，缺失时回落目的地中心点，并继续保留前端补点逻辑。

4. 前端点位种类增多后地图显得杂乱。
   - 对策：通过图例、颜色体系和按交通方式控制展示范围来降噪。

## 验收标准

满足以下条件即可认为本次需求完成：

1. 页面中餐饮被清晰拆分为 `景点附近` 和 `爆火推荐`。
2. 自驾模式下可以看到停车、加油、充电推荐，非自驾模式下不显示。
3. 地图上景点、住宿、餐饮、停车、加油、充电使用不同图标标识。
4. 点击某一天时，附近餐饮和地图路线能同步联动。
5. 历史旧数据不会因为新字段缺失而导致页面报错或主要区域空白。
